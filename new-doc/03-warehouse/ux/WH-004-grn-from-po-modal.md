### 5. Over-Receipt Validation Logic

```typescript
// Validation function
function validateReceiptQuantity(
  receivedQty: number,
  orderedQty: number,
  previouslyReceivedQty: number,
  allowOverReceipt: boolean,
  overReceiptTolerancePct: number
): ValidationResult {
  const remainingQty = orderedQty - previouslyReceivedQty;
  const totalReceivedQty = previouslyReceivedQty + receivedQty;

  // Check if over-receipt
  if (totalReceivedQty > orderedQty) {
    if (!allowOverReceipt) {
      return {
        valid: false,
        error: `Over-receipt not allowed: Max quantity is ${remainingQty} ${uom}`
      };
    }

    // Check tolerance
    const overReceiptPct = ((totalReceivedQty / orderedQty) - 1) * 100;
    if (overReceiptPct > overReceiptTolerancePct) {
      return {
        valid: false,
        error: `Over-receipt exceeds tolerance: Max ${orderedQty * (1 + overReceiptTolerancePct/100)} ${uom} (${overReceiptTolerancePct}% tolerance)`
      };
    }

    // Over-receipt within tolerance
    return {
      valid: true,
      warning: `Over-receipt of ${overReceiptPct.toFixed(1)}% (within ${overReceiptTolerancePct}% tolerance)`
    };
  }

  // Normal receipt
  return { valid: true };
}
```

## Business Rules

### Receipt Validation Rules

| Rule | Condition | Action |
|------|-----------|--------|
| **Over-receipt blocked** | received_qty + previous_received > ordered_qty AND allow_over_receipt = false | Block with error |
| **Over-receipt tolerance** | received_qty + previous_received > ordered_qty * (1 + tolerance_pct/100) | Block with error showing max allowed |
| **Over-receipt within tolerance** | received_qty + previous_received > ordered_qty BUT within tolerance | Allow with warning |
| **Batch required** | require_batch_on_receipt = true AND batch_number empty | Block with error |
| **Expiry required** | require_expiry_on_receipt = true AND expiry_date empty | Block with error |
| **Expiry in past** | expiry_date < current_date | Block with error |
| **Catch weight required** | is_catch_weight = true AND catch_weight_kg empty | Block with error |
| **Catch weight tolerance** | catch_weight_kg outside target Â±tolerance% | Warn but allow |
| **Zero quantity** | received_qty <= 0 | Block with error |
| **No lines selected** | No items in receipt array | Block with error |
| **Invalid PO status** | PO status NOT IN (confirmed, receiving) | Block with error |

### PO Status Update Rules

| Condition | New PO Status |
|-----------|---------------|
| First receipt (was 'confirmed') | 'receiving' |
| All lines fully received | 'closed' |
| Some lines partially received | 'receiving' |
| Over-receipt on any line (within tolerance) | 'receiving' + flag over_receipt_flag |

### License Plate Creation Rules

- **One LP per receipt line item** (one-to-one mapping)
- **LP number auto-generated** if `auto_generate_lp_number = true`
- **LP status**: Always 'available' initially
- **LP qa_status**: From `default_qa_status` setting (typically 'pending')
- **LP location**: User-selected location (defaults to receipt location)
- **LP inherits all batch/expiry/weight data** from receipt line

## Notes

### Design Decisions

1. **PO Selection vs Pre-selected**: Modal supports both modes - opened from PO Detail (pre-selected) or from Warehouse dashboard (selection required)

2. **Expandable Line Forms**: Each PO line can be independently expanded to add receipt details, allowing partial receipts

3. **LP Preview**: Show generated LP details before submission to help users verify data accuracy

4. **Settings-Driven Validation**: All validation rules (batch required, expiry required, over-receipt) driven by `warehouse_settings` table

5. **Over-Receipt Tolerance**: Support both strict blocking and tolerance-based over-receipt per WH-FR-029

6. **QA Status**: Default to 'pending' per settings, allow override to 'passed' for trusted suppliers

7. **Multi-Line Receipt**: Allow receiving multiple PO lines in single GRN (but each line creates separate LP)

8. **Success Modal Actions**: Provide clear next steps (view GRN, receive more, print labels) to optimize workflow

9. **Empty State**: When no POs are available for receiving, show helpful message with action buttons to guide users to create or confirm purchase orders

### Future Enhancements

- **ASN Integration**: Pre-fill receipt from Advanced Shipping Notice (WH-FR-015)
- **GS1 Barcode Scanning**: Auto-parse batch/expiry from GS1 barcodes (WH-FR-017)
- **Camera Capture**: Scan supplier delivery note photo (mobile)

- **Quality Photos**: Attach photos of goods condition during receipt
- **Weight Scale Integration**: Auto-capture catch weight from connected scale
- **Voice Input**: Voice-directed receiving workflow (WH-FR-011 scanner enhancement)
