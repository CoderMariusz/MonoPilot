## Business Rules

### Inspection Creation
1. **WO Selection**: Only WOs with status = "In Progress" or "Released" can be selected
2. **Operation Selection**: Any operation in routing can be selected (completed, in progress, or pending)
3. **Inspector Assignment**: Must be QA Inspector or Production Lead role
4. **Scheduled Date**: Cannot be in the past
5. **Test Template**: Optional, if selected, loads template parameters into Test Parameters tab
6. **Execute Immediately**: If checked, opens execution form after creation

### Inspection Execution
1. **Process Parameters**: Loaded from routing operation configuration
2. **Test Parameters**: Loaded from test template (if selected), else empty
3. **Visual Checks**: Standard checks (Appearance, Color, Texture, Consistency) always shown
4. **CCP Monitoring**: Read-only, displays CCP records from HACCP module for this WO/operation
5. **Operation Checkpoints (NEW)**: Read-only, displays checkpoint results recorded during operation execution by operators
6. **Auto-calculation**:
   - **Pass**: `measured_value >= min_value AND measured_value <= max_value`
   - **Fail**: `measured_value < min_value OR measured_value > max_value`
   - **Review**: Within 5% of min or max (borderline)
7. **Overall Result**:
   - **Pass**: All process params pass, all test params pass, all visual checks pass, no CCP deviations, no checkpoint failures
   - **Fail**: Any process param fails, any test param fails, any visual check fails, OR checkpoint failures exist
   - **Review**: No fails, but 1+ params/checks need review
8. **Failed Inspection Actions**:
   - Halt WO progression (prevent next operation from starting)
   - Create automatic quality HOLD on batch
   - Send email/SMS alert to QA Manager
   - Optional: Create NCR for investigation
   - Disposition options: Rework, Accept with Deviation, Reject, Downgrade
9. **Immutability**: Completed inspections cannot be edited (audit trail)
10. **WO Integration**: Failed in-process inspection blocks WO from progressing to next operation

### Operation Quality Checkpoints Integration (FR-QA-026)
1. **Checkpoint Results**: In-process inspection displays operation checkpoint results (read-only)
2. **Recording Location**: Checkpoints are recorded during routing operation execution by operators (not in inspection)
3. **Auto-hold on Failure**: If checkpoint marked as `auto_hold_on_failure=true`, failed checkpoint halts WO and creates hold
4. **Operator Sign-off**: Checkpoints require operator signature at time of recording
5. **Inspection vs. Checkpoint**:
   - **Checkpoints** = Tactical, per operation, operator-performed, recorded during execution
   - **Inspection** = Strategic, QA review, inspector-performed, recorded after operation
6. **Failed Checkpoint Flow**:
   - Operator records checkpoint â†’ Fail detected â†’ Auto-hold triggered (if configured) â†’ WO halted
   - QA notified â†’ Inspection created (manual or automatic) â†’ Inspector reviews checkpoint results
   - Inspector decides disposition â†’ Hold released or batch rejected
7. **Display Mode**: Operation Checkpoints tab shows results recorded by operators during operation execution (not editable in inspection)

### CCP Monitoring Integration (FR-QA-014)
1. **CCP Records**: In-process inspection displays CCP monitoring records for this WO/operation
2. **Read-Only**: CCP records managed in HACCP module, not editable in inspection
3. **Compliance Check**: Inspection shows if all CCP monitoring within limits
4. **Deviation Alert**: If CCP deviation exists, shown in inspection with link to deviation record

## API Specifications

### Inspection Management

**GET /api/quality/inspections/in-process**
- Query params: `search`, `status`, `result`, `product_id`, `date_from`, `date_to`, `sort`, `order`, `page`, `limit`
- Returns: Paginated list with WO/product/batch/operation details
- RLS: Filtered by org_id

**POST /api/quality/inspections**
- Body: `{ type: "in-process", wo_id, operation_id, scheduled_date, inspector_id, test_template_id, notes, execute_immediately }`
- Validation: WO exists, operation exists, inspector valid, scheduled_date not past
- Returns: Created inspection with ID
- Side effects: If execute_immediately=true, returns execution form data

**GET /api/quality/inspections/:id**
- Returns: Full inspection with all tabs data (process params, test params, visual checks, CCP records, checkpoint results)
- Includes: WO details, product, batch, operation, routing params, template params, checkpoint results
- RLS: Org_id match

**POST /api/quality/inspections/:id/start**
- Body: None (timestamp auto-set)
- Validation: Status must be "scheduled"
- Returns: Updated inspection with status="in_progress", started_at timestamp
- Side effects: Updates WO operation status if needed

**POST /api/quality/inspections/:id/complete**
- Body: `{ process_parameters: [], test_results: [], visual_checks: [], overall_result, final_notes, signature, disposition (if failed) }`
- Validation: All required parameters have measured values, overall result valid, signature valid
- Returns: Completed inspection with result
- Side effects:
  - Updates inspection status="complete", completed_at timestamp
  - If result="failed": Creates quality hold, sends QA Manager alert, optionally creates NCR
  - Updates WO status: If pass, allows progression to next operation; if fail, halts WO
  - Updates batch traceability record

**DELETE /api/quality/inspections/:id**
- Validation: Status must be "scheduled" (not started), user must be QA Manager or Admin
- Returns: Success message
- Side effects: Soft delete (deleted_at set)

### Related Data

**GET /api/quality/inspections/:id/process-parameters**
- Returns: Process parameters from routing operation configuration
- Includes: Parameter name, target, min, max, unit, equipment
- Source: routing_operations table, operation_quality_checkpoints table

**GET /api/quality/inspections/:id/test-parameters**
- Returns: Test parameters from template (if selected)

- Source: quality_test_templates, quality_spec_parameters tables

- Returns: CCP monitoring records for this WO
- Includes: CCP details, monitoring records, deviations
- RLS: Org_id match
- Used by: CCP Monitoring tab (read-only display)

**GET /api/quality/operation-checkpoint-results/wo/:woId**

- Source: operation_checkpoint_results table (joined with operation_quality_checkpoints)
- RLS: Org_id match
- Used by: Operation Checkpoints tab (read-only display)
- Note: Results were recorded during operation execution by operators, not during inspection

## Technical Notes

### Performance
- **Inspection List**: Index on (org_id, type='in-process', status, result, scheduled_date)
- **Execution Load**: Single query with JOINs (WO, product, batch, operation, routing, template, checkpoint results)
- **Auto-calculation**: Client-side validation (pass/fail) on blur, server-side validation on save
- **Load Time Target**: <1s for inspection list, <800ms for execution load, <2s for complete save

### Caching
- **Inspection List**: Redis key `org:{orgId}:quality:inspections:in-process:list` (30 sec TTL)
- **Inspection Detail**: Redis key `org:{orgId}:quality:inspection:{id}` (5 min TTL)
- **WO Details**: Redis key `org:{orgId}:wo:{woId}` (1 min TTL)
- **Template Parameters**: Redis key `org:{orgId}:quality:template:{id}:params` (10 min TTL)
- **Checkpoint Results**: Redis key `org:{orgId}:quality:checkpoint-results:wo:{woId}` (2 min TTL)
- **Invalidation**: On inspection create/update/delete, clear related keys

### Real-time
- **Inspection Updates**: Supabase Realtime subscription on `quality_inspections` table (type='in-process')
- **WO Status Updates**: Subscribe to `work_orders` table for status changes
- **CCP Deviations**: Real-time alerts on CCP deviation events
- **Checkpoint Failures**: Real-time alerts on checkpoint auto-hold events
- **Alert Delivery**: Edge function triggers email/SMS via SendGrid/Twilio

### Data Integrity
- **Inspection Immutability**: Completed inspections have `is_locked = true`, cannot edit
- **Audit Trail**: All changes logged to `quality_audit_log` table
- **WO Integration**: Failed inspection updates WO status to "On Hold", prevents next operation
- **Batch Traceability**: Inspection results linked to batch genealogy
- **E-signature**: Hashed with timestamp, stored in signature field
- **Checkpoint Results**: Read-only in inspection, recorded at operation execution time
